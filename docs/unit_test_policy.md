# ユニットテストポリシー

このアプリでは、開発速度と保守性を優先し、ユニットテストは以下の方針で実施する。

## 1. 目的

ユニットテストの目的は、網羅的な品質保証ではなく、以下の達成に置く。

* 主要ロジックの破壊を早期検知する
* 修正時の安心材料を確保する
* 仕様の意図をコードとして残す

## 2. テスト対象

以下に該当する処理は原則としてユニットテストを書く。

* 純粋関数
* 計算・判定・整形ロジック
* 分岐が複雑な処理
* 過去に不具合が発生した処理
* 今後も再利用される共通処理

## 3. テスト対象外

以下は原則としてユニットテストの対象外とする。

* UI の細かな見た目
* 単純な getter / setter 相当の処理
* 外部ライブラリの標準挙動そのもの
* 変更頻度が高く、テスト維持コストが高い箇所
* 画面結合でしか意味を持たない振る舞い

## 4. 粒度

* 1テスト1観点を基本とする
* 実装詳細ではなく、入力と出力に対して検証する
* モックは最小限に留める
* 壊れやすいテストは無理に維持しない
* 境界値の検証は、前後の値も含めたいわゆる三角測量的な形で行い、条件の切れ目が読み取れるようにする

## 5. 記述方針

* テストコードおよびテストケース名は日本語で記述する
* テストケース名は、平文として読んだときに期待する振る舞いが分かる表現にする
* テストケース名は、実装手順ではなく仕様や意図が伝わる書き方を優先する

## 6. 最低基準

* 新規追加する主要ロジックには、原則1件以上の正常系テストを付ける
* 分岐がある場合は、重要な異常系または境界値を1件以上追加する
* 不具合修正時は、再発防止テストを優先して追加する

## 7. カバレッジ

カバレッジの数値は参考値とし、目標達成そのものは目的化しない。
必要十分なテストがあることを優先し、形式的な網羅は追わない。

# OWダーツの旅 基本設計

## 1. 文書概要

### 1.1 目的

本書は、要件定義で整理した「OWダーツの旅」について、実装に着手できる粒度まで基本設計を整理することを目的とする。

アーキテクチャ方針（バックエンドのクリーンアーキテクチャ、フロントエンドのアトミックデザイン）は `docs/architecture.md` を参照する。

### 1.2 対象範囲

本書の対象は以下とする。

* 画面構成
* データ構造
* 状態管理
* 抽選ロジック
* localStorage 保存方式
* スロット演出の基本方針
* 主要処理フロー
* 要件を満たすための採用方針と実現方法

本書では、具体的なソースコード実装、CSS の詳細値、アニメーションの具体的な easing 数値、コンポーネント内の微細な実装手順までは扱わない。

## 2. システム構成

### 2.1 全体構成

本システムは、ブラウザ上で動作するフロントエンド単体のシングルページアプリケーションとする。

構成要素は以下の通り。

* UI 表示層
* アプリケーション状態管理層
* 抽選ロジック層
* ローカル保存層（localStorage）

### 2.2 採用想定技術

* React
* TypeScript
* Vite
* CSS または CSS Modules / Tailwind 等の軽量なスタイリング手段

### 2.3 外部依存

* バックエンドなし
* 外部 API なし
* サウンドファイルなし
* マップ画像なし（初期版）

## 3. 画面設計

## 3.1 画面一覧

初期版では単一画面構成とする。

1. メイン画面

## 3.2 メイン画面レイアウト

画面は大きく以下の領域で構成する。

### A. ヘッダー領域

* アプリタイトル
* 右上に、控えめな「使い方」導線を配置する
* 使い方押下時は、画面中央に簡潔な案内用モーダルを表示する

### B. 操作領域

* 画面左側に操作パネルを配置する
* マップルール選択 UI
  * 見出し行に「全ルールを ON に戻す」ための親チェックボックスを持つ
  * 24時間除外により候補が枯渇したルールは、自動で OFF かつ disabled 表示にする
* マップルール選択 UI の下にメッセージ表示領域を配置する

### C. スロット表示領域

* 縦スクロールの 1 リール風表示
* 中央ラインを強調する当選ライン表示
* スロット下に大型で視認性の高い抽選開始ボタンを配置する
* 抽選中・停止中は操作を受け付けない

### D. クールタイム中のマップ一覧表示領域

* 左側操作領域の下段に、マップルール選択とは別枠のパネルとして配置する
* 一覧に項目がある場合のみ、一覧ヘッダーと同じ行の右側に「CTリセット」ボタンを配置
* 24時間以内に抽選されたマップのみを新しい順に表示する
* 表示内容はマップ名のみとし、区切り記号で横並び表示してよい
* 一覧領域が画面下端に達した場合は、マップ名一覧部分のみを内部スクロールとする
* 一覧に項目がある場合のみ、「一度選ばれたマップは一定時間選ばれません」という注釈を領域上部に表示
* 「CTリセット」実行時は、一覧中のマップ名だけを短時間フェードアウトさせた後に一覧を空にする
* 上記で一覧件数が減る場合、一覧パネルの枠高さも短時間アニメーションで縮める

## 3.3 レイアウト方針

* デスクトップブラウザを前提とした横長レイアウトとする
* 左側に操作系を集約し、その中で「マップルール選択」と「クールタイム中のマップ一覧」を縦に配置する
* 中央にスロットを配置する
* 配信画面に載せた際に、中央のスロットと結果が目立つ構成とする
* 装飾は控えめにしつつ、背景・枠線・文字サイズで視認性を確保する

## 3.4 画面状態

メイン画面は以下の状態を持つ。

### 3.4.1 初期表示状態

* 保存済みデータを復元済み
* 抽選可能状態を一覧に反映済み
* 抽選開始ボタンが押下可能

### 3.4.2 抽選中状態

* スロット演出が動作中
* 抽選開始ボタンは非活性
* 「CTリセット」ボタンは非活性
* ルール選択 UI は非活性
* マッププールに存在しないルールはルール選択 UI に表示しない
* 使い方モーダルは開閉できる

### 3.4.3 結果表示状態

* 当選結果を表示
* 当選項目の強調演出を実行
* 強調演出の完了後に抽選開始ボタンは再度押下可能

### 3.4.4 候補なし状態

* 抽選開始不可
* 候補なし理由をメッセージ表示
* リセットまたはルール変更を促す

### 3.4.5 使い方モーダル表示状態

* ヘッダー右上の「使い方」導線押下で開閉する
* 同一画面内で完結し、別画面遷移は行わない
* 案内は画面中央のモーダルとして表示する
* 記載内容は「ボタンを押す」「マップが選ばれる」「オーバーウォッチを楽しむ」の流れを主軸とする

## 4. データ設計

### 4.1 マスタデータ

### 4.1.1 マップマスタ

マップ一覧は、ユーザーが用意した静的データを読み込む構成とする。
初期値は `src/data/maps.yaml` をマップマスタの定義ファイルとして読み込む。
`maps.yaml` はマップルールごとのグループを持ち、そのトップレベルキー自体を `mapRule` として採用する。
各グループ配下の `maps` 配列に各マップ定義を並べる構造とする。

#### データ項目

* mapId: string
* mapName: string
* mapRule: MapRule

#### 型定義イメージ

```ts
type MapRule = string;

type MapMaster = {
  mapId: string;
  mapName: string;
  mapRule: MapRule;
};
```

※ ルール名は実際の採用用語に合わせて調整してよい。

### 4.2 業務データ

### 4.2.1 クールタイム中のマップ一覧データ

クールタイム中のマップ一覧は、`mapState.lastSelectedAt` のうち 24 時間以内のものだけを抽選日時の新しい順に並べた派生データとして扱う。

#### データ項目

* mapId
* mapName
* selectedAt

### 4.2.2 マップ状態データ

各マップの最終選択日時を保持する。

#### データ項目

* mapId: string
* lastSelectedAt: string | null

#### 型定義イメージ

```ts
type MapSelectionState = {
  mapId: string;
  lastSelectedAt: string | null;
};
```

### 4.2.3 フィルタ状態

現在のルール選択状態を保持する。

#### 型定義イメージ

```ts
type RuleFilter = MapRule[];
```

### 4.3 画面表示用データ

画面上では、マスタデータと業務データから以下の表示用状態を算出する。

### 4.3.1 マップ表示状態

```ts
type MapAvailabilityStatus =
  | 'available'
  | 'excluded-by-24h'
  | 'excluded-by-filter';
```

### 4.3.2 表示用マップ情報

```ts
type MapDisplayItem = {
  mapId: string;
  mapName: string;
  mapRule: MapRule;
  displayLabel: string; // `${MapRuleLabel} : ${mapName}`
  status: MapAvailabilityStatus;
  availableAt?: string | null;
};
```

## 5. 保存設計

### 5.1 保存先

保存先は localStorage とする。

### 5.2 保存キー

* `ow-darts-journey.mapState`
* `ow-darts-journey.ruleFilter`

キー名は将来的な拡張を考慮し、アプリ識別子を付ける。

例:

* `ow-darts-journey.mapState`
* `ow-darts-journey.ruleFilter`

### 5.3 保存内容

### 5.3.1 mapState

各マップの最終選択日時を保存する。
保存時は JSON 文字列として localStorage に格納する。

```ts
type StoredMapState = Record<string, string | null>;
```

### 5.3.2 ruleFilter

最後に使用したルール選択状態を保存する。
保存時は JSON 文字列として localStorage に格納する。

```ts
type StoredRuleFilter = RuleFilter;
```

日時文字列は、扱いやすさと可搬性を考慮して ISO 8601 形式を採用する。

### 5.4 復元方針

* アプリ起動時に localStorage から保存データを読み込む
* 保存データが存在しない場合は初期値を使用する
* 不正な JSON や型不整合がある場合は破棄し、初期値にフォールバックする

## 6. 状態管理設計

### 6.1 主要 state

アプリケーション状態として、少なくとも以下を管理する。

* 抽選候補一覧
* 候補なし判定
* クールタイム中のマップ一覧表示
* リール表示用データ

## 7. 抽選ロジック設計

### 7.1 抽選条件

抽選候補は以下の順で絞り込む。

1. マップマスタ全件を起点とする
2. チェックされたマップルールを適用する
3. 24時間除外を適用する
4. 残件から一様乱数で 1 件選択する

### 7.2 24時間除外判定

* 現在時刻と lastSelectedAt を比較する
* 経過時間が 24 時間未満の場合は除外とする
* 24 時間以上経過していれば候補に戻す

### 7.3 候補なし判定

候補件数が 0 件の場合、抽選は実行しない。

表示メッセージは以下の優先順で構成する。

* 24時間除外により候補が存在しない
* チェックされたルールに候補が存在しない

### 7.4 抽選後更新

抽選確定後、以下を更新する。

1. 選択マップの lastSelectedAt を現在時刻で更新
2. 当選した mapRule のチェックを外す
3. 最後の1ルールだった場合は、当選した mapRule のみ OFF にし、それ以外のマッププール内に存在する全ルールを ON にする
4. localStorage に保存
5. 画面表示を更新

## 8. スロット演出設計

### 8.1 基本方針

* 縦スクロールの 1 リール風表示とする
* 実際の抽選結果は、演出開始前に内部的に確定してよい
* リール演出は視覚的演出として実装する

### 8.2 リール表示内容

各リールに流れる項目は以下の形式とする。

* `RULE : MAP NAME`

例:

* `HYBRID : KING'S ROW`

リール表示用データは、現在の ruleFilter や 24 時間除外とは独立して、マップマスタ全件から構成する。

リール内の並び順は画面読込時に一度だけランダム化し、同一ルールのマップが連続しすぎない見え方を許容する。

一方で、実際の当選結果は 7.1 の抽選候補生成結果からのみ選択する。

### 8.3 リール構成

* 1 つの縦リールを表示する
* 中央位置を当選ラインとして視覚的に強調する
* 上下の表示領域は中央よりも狭く見せ、フェードやぼかしで視線を弱める
* 中央ライン付近の当選文字列は大きめの文字サイズで表示し、周辺候補より視認性を高める
* 最終停止時は、中央ラインに当選結果が揃って見える構成とする

### 8.4 アニメーションの流れ

標準演出時間は 2.8 秒とする。

大まかな流れは以下とする。

1. 開始直後に素早く加速
2. 早い段階で最高速に到達する
3. 最高速で一定時間スクロールする
4. 終了前に減速する
5. 当選位置でピタッと停止する
6. 停止した項目を短時間強調する
7. 結果表示中も、停止したリール表示はそのまま残す

開始直後の加速区間は短めに取り、その後は複数周ぶんのスクロール量を確保して、最高速時は個々の候補を追いづらい見た目とする。

### 8.5 強調演出

停止した瞬間、当選文字列に対して以下のいずれか、または組み合わせを行う。

* 軽微な拡大
* 短い前方移動
* 短いバウンス
* 影や発光による強調
* 停止後にごく短い間を置いてから入る発光

※ 具体値は詳細設計で決定する。

### 8.6 注意点

* 演出は派手すぎず、文字の可読性を優先する
* 1 リール風であっても、結果は明確に 1 件と認識できること
* 演出中の連打や二重起動は防ぐ

## 9. 処理フロー設計

### 9.1 起動時フロー

1. マップマスタを読み込む
2. localStorage から保存データを復元する
3. 不正データがあれば初期値にフォールバックする
4. 現在時刻を基準に各マップの状態を算出する
5. 画面に表示する

### 9.2 抽選開始フロー

1. 現在の ruleFilter（チェック状態）を取得する
2. 抽選候補一覧を生成する
3. 候補が 0 件ならメッセージ表示して終了する
4. 候補から 1 件をランダムに確定する
5. slotStatus を `spinning` に変更する
6. リール演出を開始する
7. リール停止後、強調演出の完了を待って結果表示へ遷移する

### 9.3 抽選終了フロー

1. 抽選結果で mapState を更新する
2. 当選した mapRule を ruleFilter から外し、最後の 1 つだった場合は当選ルールのみ OFF、それ以外の利用可能ルールを ON にする
3. localStorage に保存する
4. slotStatus を `stopping` に変更する
5. 強調演出完了後に slotStatus を `result` に変更する
6. 停止したリール表示を維持したまま、結果表示・一覧表示を更新する

### 9.4 リセットフロー

1. ユーザー確認を取得する
2. mapState を初期化する
3. ruleFilter を表示中の全ルール ON に戻す
4. localStorage を更新する
5. 画面表示を更新する

## 10. 例外・異常系設計

### 10.1 保存データ破損

* JSON パース失敗時は保存データを無視する
* 初期値で起動し、必要ならメッセージ表示する

### 10.2 候補ゼロ

* 抽選を開始しない
* 理由メッセージを表示する

### 10.3 抽選中の再操作

* 抽選開始ボタンを無効化する
* 「CTリセット」ボタンを無効化する
* フィルタ変更を無効化する

### 10.4 マップマスタ不正

* mapId 重複や必須項目欠落がある場合、エラーとして扱う
* 初期版ではコンソールエラー出力とし、必要に応じて画面メッセージも検討する

### 補足

* `data/` にマップマスタを配置する

* `docs/` に要件定義と設計書を置く
